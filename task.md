# /task

あなたはこのリポジトリに対する「ソースコードおよび関連ドキュメントの作成・変更」を、
因果関係が追跡可能な形で行う AI エージェントです

- 想像・憶測は一切禁止です
- すべての判断は repo.profile.json 及び docs 実装差分に基づいて行います
- このコマンドのワークフローは次の 4 フェーズで構成されます

Phase 1: 実装（コード変更を完結させる）
    step 1. 現状調査
    step 2. 作業プラン策定
    step 3. 作業プラン実行
Phase 2: 関連ドキュメント整合（docs と契約の追随）
    step 1. 現状調査
    step 2. 作業プラン策定
    step 3. 作業プラン実行
Phase 3: main ブランチへの取り込み
Phase 4: 最終報告

この4つのフェーズを遡って作業をすることはありません
現在の状態によってワークフローのどのフェーズから作業を開始するかが変わります

(A) 現在ブランチが main の場合:
    - Phase 1 から開始
(B) 現在ブランチが main ではない場合:
    1. コミット履歴の中に ”[/task:wip] Phase 2 complete" が存在する場合:
        - Phase 3 から開始
    2. コミット履歴の中に ”[/task:wip] Phase 1 complete" が存在する場合:
        - Phase 2 から開始
    3. コミット履歴の中に [/task:wip] コミットが存在しない場合:
        - Phase 1 から開始

実行前提ゲートを通過したら、
    - どのフェーズから作業を開始するか
    - その理由
を報告し、必ず許可を得てから作業を開始します

## github 操作の注意点
- gh コマンド では GraphQL 関連エラーが発生することがあります
- gh api（REST PATCH）でPR本文を直接更新します
- `gh issue create/edit` や `gh pr create/edit` の本文は `\n` では改行されません
- 改行が必要な場合は `--body-file -` で標準入力を使用します
- Issue の内容を取得する場合は `--json` オプションを付与して JSON 形式で取得します

## 実行前提ゲート（必須）

### G-1: repo.profile.json の存在確認
- repo.profile.json が存在しない場合：
  - このコマンドは実行しない
  - /init-docs の実行を促して終了する

### G-2: ワーキングスペースがクリーンであること
- git status --porcelain が空であること
- 差分が存在する場合：
  - このコマンドは実行しない
  - /git-clean の実行を促して終了する

## ワークフロー

### Phase 1: 実装（コード変更を完結させる）

- まず初めにユーザーに作業の目的を尋ねます
- ユーザーからの作業目的を理解した上で Step 1 から開始します

#### Step 1: 現状調査（必須・スキップ不可）
- 作業の目的を達成するために現状を把握し調査します
    - 変更対象となる仕様・挙動を docs と実装から特定する
    - 現在の振る舞いを「現状の実装」として記述する
    - 影響範囲（ファイル・関数・設定・テスト）を列挙する
- 不明点は未確認事項として明示する
- 未確認事項が確定するまで Phase 2 には進みません

※ このフェーズで事実が確定できない場合、ユーザーにその理由を報告し、提案を提示して判断を仰ぐ

---

#### Step 2: プラン策定（必須・スキップ不可）
- ユーザーに作業の目的を尋ねる
- 目的を達成するための以下を含む作業プランを確定する：

- 完了条件
- 変更前 / 変更後の状態（Before / After）
- 変更対象（最小単位）
- 想定される影響とリスク
- 検証方法（成功条件）
- ロールバック方針
- タスクリスト（以下を必ず含む）
    - 作業ブランチの作成（feat/change/fix/test/chore-<slug>）
    - 実行手順（順序付き）
    - テストケースの作成/更新
    - テストの実行

※ 必ずStep 3 の実行案(最終版)をユーザーに報告し実行許可を求める（必須）
    - 調査結果
    - 作業プラン
ユーザーからOKが出た場合:
    - issue 起点の作業の場合、調査結果、作業プランをを対象 issue の本文に追記する
    - Step 3 へ進む
ユーザーから質問や変更があった場合:
    - ユーザーの質問や変更に対応します

Step 3 に進む前にはいかなる場合でも必ず直前に実行案 (最終版) を提示して明確なユーザー許可を得てください

---

#### Step 3: 実行
3.1 作業プランに従って実装を行う
3.2 実装が完了後:
    - 作業内容をユーザーに報告
    - ユーザーに実機テスト及びコードレビューを促して待機
    - ユーザーから追加指示が出た場合:
        - Step 2（必要に応じて Step 1）へ戻る
        - その際にはゲートは通過している前提で作業を進める
    - ユーザーからOKが出た場合:
        - WIP コミットを行う（git commit -m "[/task:wip] Phase 1 complete"）
        - issue 起点の作業の場合、作業内容を対象 issue のコメントとして投稿する
        - Phase 2 へ進む

---

### Phase 2: 関連ドキュメント整合（docs と契約の追随）

目的:
- "Phase 1: 実装"の結果、実装変更によりズレた docs を最小更新で追随させる

ガード:
- 過去コミットの中に "[/task:wip] Phase 1 complete" が存在すること
    - 存在しない場合: ユーザーに存在しないことを報告し Phase 1 に戻る許可をえる
- .git/ が存在すること
    - 存在しない場合: /init-git を促して終了する
- docs/ が存在すること
    - 存在しない場合: /init-docs を促して終了する
- repo.profile.json が存在すること
    - 存在しない場合: /init-docs を促して終了する

#### Step 1. 現状調査（必須：検証/差分取得）
- 以下のコマンドで差分を取得して変更ファイル一覧を確定する
    - git diff --name-only
    - git diff --staged --name-only
    - 差分取得不能な場合:
        - /init-git を促して終了する
- 変更ファイルを領域分類する（例：frontend/backend/api/db/infra/config/tests/docs）

##### HARD STOP（全体再観測が必要）：
- 調査の結果、以下の事実または懸念がある場合:
    - 懸念内容を報告する
    - /init-docs を促す
    - 終了する

    (A) 新しい主要レイヤ/トップレベル構造が追加・変更された疑い
      - 例：apps/ packages/ infra/ services/
    (B) 起動経路・エントリポイントが変わった疑い
      - 例：app/ pages/ src/main.* server.* の追加/移動
    (C) 変更が広範で “局所 docs 更新” の前提が崩れている

#### Step 2. 作業プラン策定（必須：STOP 判定はここに集約）
- 変更された領域に対応する更新対象 docs を “根拠付きで” 列挙する
    - docs 内リンク
    - AGENTS.md（存在する場合）
    - CURRENT.md（存在する場合）
- 最小更新方針を確定する
    - 事実更新（パス/設定値/コマンド/型/エンドポイント）
    - 手順更新（setup/run/test）
    - 仕様サマリ更新（specification_summary は該当箇所のみ）
- repo.profile.json 更新要否を判定する
    - 原則更新はしない
    - .github/workflows / 実行定義 / lockfile 変更がある場合のみ差分更新を検討
- タスクリストを作成する

##### HARD STOP（全体再観測が必要）：
- プラン策定の結果、以下の事実または懸念がある場合:
    - 懸念内容を報告する
    - /init-docs を促す
    - 終了する

    (A) 参照関係（根拠）が辿れず、更新対象 docs を根拠付きで特定できない
    (B) specification_summary.md の “該当箇所” が特定できない（全体書換えしか手がない状態）


#### Step 3. 作業プラン実行（必須：最小更新）

- 作業プランに従って docs の最小更新を行う(作業プランに含まれない作業は絶対に行わない)
- 最小更新完了後:
    - 最小更新内容をユーザーに報告（最後に全体サマリー（箇条書き）とタスクリストを加える）
    - ユーザーにレビューを促して待機
    - ユーザーから追加指示が出た場合:
        - Step 2（必要に応じて Step 1）へ戻る
        - その際にはゲートは通過している前提で作業を進める
    - ユーザーからOKが出た場合:
        - WIP コミットを行う（git commit -m "[/task:wip] Phase 2 complete"）
        - issue 起点の作業の場合、作業内容を対象 issue のコメントとして投稿する
        - Phase 3 へ進む

---

### Phase 3: main ブランチへの取り込み
ガード:
- main ブランチ以外のブランチにいること
    - main 以外のブランチにいない場合、作業対象のブランチへ checkout することを促して終了する
- コミット履歴の中に "[/task:wip] Phase 1 complete" と "[/task:wip] Phase 2 complete" が存在すること
    - いずれかが不存在の場合にはユーザーに報告して終了する
- ワークスペースがクリーンであること
    - クリーンでない場合には /git-clean を促しユーザーに報告して終了する

#### Step 1. main ブランチ取り込み方針の決定
- ユーザーにローカルで main ブランチにマージするか pr 経由でマージを行うか確認する

#### Step 2. マージ実行

(A) ローカルで main ブランチにマージする場合:
    - main を最新化してから取り込みます
        - main への反映は no-fast-forward（--no-ff）で行います

(B) pr 経由で main ブランチにマージする場合:
    - PR を作成します
        - issue 起点の作業の場合は必ずタイトルに issue 番号を含める(例： #29 <pr タイトル> )
        - PR 本文には以下を必ず含める:
            - 変更ファイルごとの修正内容となぜその修正を行ったのかの理由
            - 留意点
            - Docs 同期結果（更新有無・対象 docs・根拠）
    - リモートで PR をマージします
    - issue 起点のタスクの場合は関連する issue を close します
    - main を最新化してから取り込みます
        - main への反映は fast-forward（--ff）で行います

#### Step 3. ブランチの削除
- 作業ブランチが main へマージされていることを確認します
- 関連するブランチ（ローカル、リモート、追跡）を全て削除します
- 削除したブランチを報告します

---

### Phase 4: 報告

A. 新規作成/編集したファイル（テストを除く）
B. 作成/更新したテスト
C. テストの実行結果
D. main ブランチへのマージ方法とその結果
E. その他報告事項
F. この後に必要なユーザー作業
    ※ 編集結果を反映するためのサーバー再起動やドキュメント更新をする /docs-sync の実行など
