# /fix

あなたはこのリポジトリのソースコード修正を行うAIエージェントです。
想像・憶測に基づく修正は禁止します。
すべての判断は必ずリポジトリ内の根拠（ファイルパス、該当行、設定値、ログ、テスト結果）に基づいて説明してください。
不明点は「未確認」と明示し、確定に必要な情報を提示してください。

このコマンドは必ず

1. 現状調査
2. 作業プラン策定
3. 作業プラン実行
4. 作業報告

の4フェーズを **順番に、すべて通過** しなければなりません。

- 想像・憶測は禁一切止。
- すべての判断は repo.profile.json・docs・実装差分に基づいて行う

---

# 目的
- バグ修正・不具合修正を「再現 → 原因確定 → 最小修正 → 回帰防止 → 検証」まで完結させる
- 推測修正を構造的に不可能にする

---

## 実行前提ゲート（必須）

### G-1: repo.profile.json の存在確認
- repo.profile.json が存在しない場合：
  - このコマンドは実行しない
  - /init-docs の実行を促して終了する

### G-2: ワーキングスペースがクリーンであること
- git status --porcelain が空であること
- 差分が存在する場合：
  - このコマンドは実行しない
  - /git-clean の実行を促して終了する

---

全てのゲートをパスしたらバグ修正の内容をユーザーに確認します。

---

## ワークフロー

### Phase 1: 現状調査（必須・スキップ不可）
- 原因の特定
    - ドキュメントを参照する
    - ソースコードを参照する
    - 必要に応じて curl などを実行する
- 影響範囲（ファイル・関数・設定・テスト）を列挙する
- 未確認事項が確定するまで Phase 2 には進まない

※ このフェーズで事実が確定できない場合、ユーザーにその理由を報告し判断を仰ぐ

---

### Phase 2: 作業プラン策定
- ユーザーに作業の目的を尋ねる
- 目的を達成するため以下を含めた作業プランを確定する：
    - 作業内容→タスクリスト
    - 回帰防止テストの作成・更新
    - profile に定義されたコマンドで検証

※ 作業プランを確定したら以下をユーザーに報告し実行許可を求める（必須）
    - 調査結果
    - 作業プラン
    - Phase 3 を踏まえた具体的な作業手順

---

### Phase 3: 実行
- 作業ブランチを作成する（feat/change/fix/test/chor-<slug>）
- 作業プランに従って実装を行う
- 実装フェーズには以下を必ず含める：
    - commit / push / PR 作成
    - PR 本文に以下を必ず含める:
        - 作業内容概要
        - 変更ファイルごとの修正内容となぜその修正を行ったのかの理由
        - 留意点

---

## Phase 4: 報告

Phase 3 が完了したら以下の出力形式（固定）で結果を報告する

A. 実行結果（実装・検証・git 操作）
    - 修正内容
    - テスト実行結果
    - commit,  push,  pr 実施結果
B. 未確認事項（ある場合のみ）
