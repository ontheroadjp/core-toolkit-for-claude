# test-balance.md

## 概要（Purpose）

`test-balance` は、リポジトリ全体を対象に
**「テストが必要な領域」と「実際のテスト状況」** の整合性・バランスを維持するためのコマンドである。

本リポジトリでは日々のソース改変は `/task` が担い、`/task` 内で変更範囲に応じたテスト追加/更新を行う。
その前提のもと `test-balance` は、`/task` だけでは拾いにくい **横断的ギャップ（非局所）** を検出し、
**次に `/task` で実装すべき“テスト改善タスク”の作業指示書（Task Brief）を生成**する。

- `test-balance` 自体は原則としてソース/テストを直接編集しない（改変経路を `/task` に統一）
- 不足が疑われる箇所を **高・中・低**で分類し、proxy指標で不足判定する
- 実装はユーザーが候補を選択し、その選択に基づいて `/task` 用のブリーフを出力する

---

## 基本方針（Principles）

- 想像・推測で判断しない（観測可能な根拠に基づく）
- “テストが必要”を一律に扱わない（高・中・低の3段階）
- カバレッジ率を直接の判断材料にしない（運用コストが高く誤差も大きい）
- 暴走防止：提案数は最大3、ブリーフ生成も最大3に制限
- 改変経路は `/task` に一本化し、`test-balance` は「診断＋ブリーフ生成」に徹する

---

## フェーズ構成

### Phase 1: 領域分類（Classification）
ソースコードを走査し、各ファイル/モジュールを
**テスト必要度：高・中・低**に分類する。

### Phase 2: テスト存在判定（Coverage Proxy）
分類結果に対して、対応するテストが「存在するか / 不足しているか」を
**proxy 指標**により判定する。

### Phase 3: 改善候補提示（Suggestions）
不足（Weak / Missing）が疑われる箇所について
**最小単位の改善候補**を番号付きで提示する（最大3件）。

### Phase 4: /task ブリーフ生成（Task Brief Generation）
ユーザーが選択した候補について、`/task` に渡すための
**作業目的・完了条件・影響範囲・検証・ロールバック・タスクリスト**を含むブリーフを生成する。

※ `test-balance` はここでコード変更を実行しない。実装は `/task` で行う。

---

## Phase 1: テスト必要度の3段階分類

### 高（High Priority）
「失敗時の影響が大きく、テストで守られるべき領域」

#### 該当項目
- ビジネスロジックの中核（料金/権限/状態遷移/整合性）
- 外部 I/O 境界（HTTP / API / DB / File / Queue 等）
- 認証・認可・権限判定
- 入力バリデーション・スキーマ変換
- セキュリティ関連処理
- 永続化・トランザクション・マイグレーション

#### 観測シグナル例（推測禁止）
- 外部 SDK / API / DB クライアントの使用
- `fs` / `http` / `fetch` / `axios` / request 等の利用
- `routes` / `controllers` / `handlers` / `usecase` / `domain` 配下
- `auth` / `permission` / `role` / `token` / `session` 等の語彙
- `schema` / `validate` / `parse` / `normalize` 等の処理

---

### 中（Medium Priority）
「致命傷ではないが、品質安定に寄与する領域」

#### 該当項目
- ユースケース層・サービス層
- データ整形・集計・変換ロジック
- エラーハンドリング・リトライ処理
- CLI / バッチ処理の主要フロー
- 複数箇所から再利用されるユーティリティ

#### 観測シグナル例
- `service` / `interactor` / `processor` 等の配置
- 複数 import 元を持つ（再利用の痕跡）
- 直近変更頻度が高いファイル（ホットスポット）

---

### 低（Low Priority）
「テスト必須ではなく、静的チェックで十分な領域」

#### 該当項目
- 一発ユーティリティ・開発補助スクリプト
- 表示・ログ整形のみの処理
- 変更頻度が極めて低いコード
- CI / 実行経路から参照されない補助コード

#### 観測シグナル例
- `tools/` `scripts/` `bin/` `dev/` `misc/` 配下
- 実行権限が無い、かつ参照経路が観測できない
- CI や entrypoint から呼ばれていない

---

## Phase 2: テスト存在判定（Coverage Proxy）

厳密なカバレッジ測定は行わず、以下を **proxy 指標**として使用する。

### 判定基準
- 近傍にテストファイルが存在する
  - `tests/`, `__tests__/`, `test/`, `*.test.*`, `*.spec.*`
- テストコード内で対象モジュールが import / require されている（粗くてよい）
- 高優先度領域は最低限：
  - 正常系の通過テスト、または
  - 異常系（不正入力/エラー）のテスト
- I/O 境界は最低限：
  - 外部依存をモックし主要パスを通すテスト（統合でなくてもよい）

### 判定結果
- OK：十分
- Weak：存在するが不足の可能性が高い
- Missing：存在しない（近傍にも参照にも痕跡がない）

---

## Phase 3: 改善候補の提示（最大3件）

不足（Weak / Missing）が検出された場合、
**最小単位の改善候補**を次の形式で提示する（最大3件に制限）。

### 出力形式（例）
1. [High] src/payment/calculate.ts
   - 根拠: 金額計算ロジック（コア）
   - 現状: 異常入力のテストが見当たらない（Weak）
   - 最小改善案: 異常入力1ケース + 境界値1ケースの unit test 追加

2. [High] src/api/user_handler.ts
   - 根拠: HTTP handler（I/O境界）
   - 現状: 主要パスを通すテストが見当たらない（Missing）
   - 最小改善案: service層をモックして 200/400 の2ケースのテスト追加

3. [Medium] src/user/service.ts
   - 根拠: 再利用モジュール + 変更頻度高（ホットスポット）
   - 現状: importはあるが assertion が弱い（Weak）
   - 最小改善案: 主要分岐1つをカバーするテスト追加

---

## Phase 4: /task ブリーフ生成（実装は /task で行う）

ユーザーが Phase 3 の候補番号を指定したら（例：「1 と 3」）、
指定された各候補について `/task` 用の **Task Brief** を生成する。

### ブリーフ生成ルール
- 生成するブリーフは最大3件（候補の上限に一致）
- `/task` の Step 2（プラン策定）に必要な要素を満たす
  - 完了条件 / Before-After / 変更対象 / 影響とリスク / 検証 / ロールバック / タスクリスト（テスト作成/更新・テスト実行を含む）
- 実装は `/task` が担う。`test-balance` はコード編集を実行しない。
- 未確認事項がある場合は、ブリーフ内で「未確認」と明示し、観測手順を含める。

---

## /task Task Brief 出力フォーマット（固定）

以下を各候補について出力する（そのまま `/task` に貼れる粒度）。

### Task Brief（テンプレ）

- タイトル：
- 目的（Why）：
- 完了条件（Done Criteria）：
- Before / After（期待する変化）：
- 変更対象（最小単位）：
  - 対象ファイル:
  - 対象関数/モジュール（分かる範囲）:
- 想定される影響とリスク：
- 検証方法（成功条件）：
  - 実行するコマンド（repo.profile.json の commands を参照）：
  - 期待結果：
- ロールバック方針：
- タスクリスト（必須項目を含む）：
  1) 作業ブランチ作成（例：test/<slug>）
  2) 実行手順（順序付き）
  3) テストケースの作成/更新（ここが主）
  4) テストの実行（verify/test/typecheck/lint の該当分）
- 未確認事項（ある場合のみ）：
  - 未確認点：
  - 確定に必要な観測：

---

## Final Report（固定）

- A. 分類結果サマリ（High / Medium / Low 件数 + 主な根拠）
- B. 不足判定（OK / Weak / Missing の一覧）
- C. 改善候補（最大3件、番号付き）
- D. ユーザー選択（例：1,3）
- E. /task Task Brief（選択分のみ）
- F. 未確認事項（ある場合のみ）

---

## 本コマンドの位置づけ

- `init-test`：テスト基盤を整える（commands・smoke・CI整合の入口）
- `/task`：日々の改変を行い、その変更範囲のテストを最小追従する
- `test-balance`：横断的ギャップを検出し、次に `/task` で実装すべき改善タスクを生成する

`test-balance` は
「テストを増やすためのコマンド」ではなく、
「必要な場所だけに、必要な分だけ投資するための診断＋指示書生成コマンド」である。
