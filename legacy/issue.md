# /Issue

## Role
Issue を「実行可能な作業単位」に正規化する。
/issue コマンドから参照され、本文品質を保証する。

---

## ワークフロー

### Phase 1: 壁打ち(Wall-Chat)

まず最初に作業（タスク）の内容をユーザーに確認します。

#### Step 1: 現状調査（必須・スキップ不可）
- 変更対象となる仕様・挙動を docs と実装から特定する
- 現在の振る舞いを「現状の実装」として記述する
- 影響範囲（ファイル・関数・設定・テスト）を列挙する

※ このフェーズで事実が確定できない場合、ユーザーにその理由を報告し、提案を提示して判断を仰ぐ

---

#### Step 2: 壁打ち（Wall-Chat）
- ユーザーとの壁打ちを行います
- 不明点は未確認事項として明示する
- 未確認事項がある場合は Phase 2 には進みません
- ユーザーからの承認が出るまで Phase 2 には進みません
- ユーザーからの承認が出るまで壁打ちを継続します

### Phase 2: Issue の起票

- Issue は類似 Issue を複数作成しません

#### Step 1: Issue 作成方針の確定
- 既存のオープン Issue 一覧を取得します
- Issue をどのような形式で起票することが最も適しているのか、理由とともにユーザーに提案します
- Issue 形式を以下の2つの形式から最良の形式を判断して提示します
    A) 単一 Issue: ひとつの Issue
    B) 親子 Issue: 親 Issue とそれに紐づく複数の子 Issue
- 複数の単一 Issue とするのか親子Issue とするのかは、作業間に概念的な関係性があるかどうかです
- Issue 起票には次の5パターンがあります
    A) 既存 単一 Issue に組み込む（既存 Issue のアップデート）※ この場合には新たな起票はしません
    B) 既存 単一 Issue を親子 Issue に再編して起票
    C) 既存 親子 Issue の子 Issue として起票
    D) 独立した単一 Issue として起票
    E) 独立した親子 Issue として起票
- ユーザー承認が得られたら step 2 へ進みます
- ユーザー承認が無い限り step 2 へは進みません

#### Step 2: Issue の起票
- Step 1 で承認された形式で Issue を作成します
- Issue の起票には gh コマンドを使用します

※  Parent Issue: #<Issue番号> を明記すること
※  Sub Issue 一覧は、- [ ] #<Issue番号> の形式（タスクリスト）で記述すること
※  計完了条件は、測可能、再現可能、第三者が読んでも検証可能であること
※  1次調査メモには Phase 1 で調査した結果を記述

---

##### Issue 本文テンプレ
単一 Issue には以下の5項目を含めます。

1. 背景
2. 対象
3. 目的
4. 完了条件
5. 1次調査メモ

親 Issue には以下の4項目を含めます。

1. 背景/目的
2. スコープ
3. Sub Issue 一覧
4. 1次調査メモ

子 Issue には以下の5項目を含めます。

1. 背景
2. 親Issue
3. 目的
4. 完了条件
5. 1次調査メモ
