# /task

あなたはこのリポジトリに「テスト基盤」を整備するAIエージェントです。
想像・憶測は禁止。すべての判断は必ずリポジトリ内の根拠（設定ファイル、既存CI、実行結果）に基づいて説明してください。
不明点は「未確認」と明示し、確定に必要な観測やファイルを提示してください。

このコマンドは必ず

1. 現状調査
2. 作業プラン策定
3. 作業プラン実行
4. 作業報告

の4フェーズを **順番に、すべて通過** しなければなりません。

- 想像・憶測は禁一切止。
- すべての判断は repo.profile.json・docs・実装差分に基づいて行う

---

## 実行前提ゲート（必須）

### G-1: repo.profile.json の存在確認
- repo.profile.json が存在しない場合：
  - このコマンドは実行しない
  - /init-docs の実行を促して終了する

### G-2: ワーキングスペースがクリーンであること
- git status --porcelain が空であること
- 差分が存在する場合：
  - このコマンドは実行しない
  - /git-clean の実行を促して終了する

---

全てのゲートをパスしたら作業（タスク）の内容をユーザーに確認します。

---

## 目的
- プロジェクト進行段階に応じたテストケースを作成する

## ワークフロー

### Phase 1: 現状調査（必須・スキップ不可）
- 変更対象となる仕様・挙動を docs と実装から特定する
- 現在の振る舞いを「現状の実装」として記述する
- 影響範囲（ファイル・関数・設定・テスト）を列挙する
- 不明点は未確認事項として明示する
- 未確認事項が確定するまで Phase 2 には進まない

※ このフェーズで事実が確定できない場合、ユーザーにその理由を報告し、提案を提示して判断を仰ぐ

---

### Phase 2: 作業プラン策定
- 目的を達成するための以下を含む作業プランを確定する：

- 完了条件
- 変更前 / 変更後の状態（Before / After）
- 変更対象（最小単位）
- 実行手順（順序付き）
- テストケースの作成/更新
- テストの実行
- 想定される影響とリスク
- 検証方法（成功条件）
- ロールバック方針

※ 作業プランを確定したら以下をユーザーに報告し実行許可を求める（必須）
    - 調査結果
    - 作業プラン
    - Phase 3 を踏まえた具体的な作業手順

---

### Phase 3: 実行
- 作業ブランチを作成する（feat/change/fix/test/chor-<slug>）
- 作業プランに従って実装を行う
- 実装が完了したら作業内容をユーザーに報告をしてテストを促す
- ユーザーから追加指示が出れば Phase 2（必要に応じて Phase 1）へ戻る。その際にはゲートは通過している前提で作業を進める
- ユーザーからOKが出れば、
    - commit / push / PR 作成
    - PR 本文には以下を必ず含める:
        - 作業内容概要
        - 変更ファイルごとの修正内容となぜその修正を行ったのかの理由
        - 留意点

---

## Phase 4: 報告

Phase 3 が完了したら以下の出力形式（固定）で結果を報告する

A. 新規作成/編集したファイル（テストを除く）
B. 新規作成/編集したファイル（テストのみ）
C. テストの実行結果
D.  commit,  push,  pr 実施結果
E. その他報告事項
F. この後に必要なユーザー作業
    ※ 編集結果を反映するためのサーバー再起動やドキュメント更新をする /docs-sync の実行など
