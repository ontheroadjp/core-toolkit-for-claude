# init-docs.md

あなたはこのリポジトリの「実態に基づく」プロジェクト設計ドキュメントを作成・更新・管理するAIエージェントです。
想像・憶測は禁止。すべての主張は、必ずリポジトリ内の根拠（ファイルパス、該当行、設定値、実装箇所）に紐づけて説明してください。
不明な点は「未確認」と明示し、追加で何を見れば確定できるかを提示してください。

---

## 目的
- AI がこのリポジトリを一貫した前提で扱える状態を作る
- repo.profile.json を「契約（Single Source of Truth）」として確立する
- 設計ドキュメントと実装の対応関係を再構築する

---

## 実行方針（最重要）
- まず現状分析 → Repo Profile 生成 → ドキュメント生成 → AGENTS.md 更新
- ドキュメントは「なぜそうなっているか」を必ず説明する
- 冗長な百科事典は禁止。根拠と決定事項を中心に書く
- 生成後、必ず差分サマリ（新規 / 更新 / 保留）を出す

---

## Phase 0.5: 再実行トリガー（運用ルール）
以下のいずれかに該当する場合、
このコマンドを実行して **再観測・再構築**を行うこと。

- docs-sync が HARD STOP を検知した
- repo.profile.json に載っていないコマンドや doc_root が運用上必要になった
- docs（L1/L2/L3）が現状の構造や責務を説明できなくなった
- 起動方法・主要エントリポイントが変わった疑いがある
- 新しいレイヤ（apps / packages / infra 等）が導入された

---

## Phase 1: プロジェクト分析
- 技術スタック特定
- ディレクトリ構造と責務の把握
- 実装済み機能の検出
- 主要依存関係の裏取り

すべて根拠付きで列挙すること。

---

## Phase 1.5: Repo Profile 生成（repo.profile.json）
- 想像・補完は禁止
- 確定できない項目は出力しない
- CI 定義 > 実行定義 > docs の順で根拠を採用
- 既存ファイルがある場合は差分更新のみ

最小スキーマ例（参考）：

    ```json
    {
      "repo_id": "string",
      "package_manager": "pnpm|npm|yarn|bun|unknown",
      "doc_roots": [],
      "commands": {}
    }
    ```

## Phase 2.5: 整合性検証（必須・保証フェーズ）

このフェーズの目的は、
「生成したドキュメントが、現時点のソースコード・設定・CI 定義と
**機械的に矛盾していないこと**を確認し、保証できない部分を明示的に切り分けること」である。

ここを通過しない限り、
/init-docs は「整合性が取れた」と宣言してはならない。

---

### 基本原則
- 想像・補完は禁止
- 「合っているはず」は不可
- 確認できないものは **必ず未確認事項として分離**
- 暗黙の前提を docs に残さない

---

### 2.5-1. docs → 実体 の検証（必須）

生成・更新した docs 内から、以下をすべて抽出し検証する。

#### 検証対象
- ファイルパス
- ディレクトリパス
- コマンド（run / build / test / dev 等）
- エントリポイント（main, app, server 等）
- 設定ファイル参照（config, env, CI）

#### 検証ルール
- パス：実在すること（大文字小文字含む）
- コマンド：
  - repo.profile.json の commands に存在する
  - または CI 定義（.github/workflows）に存在する
- エントリポイント：
  - 実ファイルが存在し、docs の説明と矛盾しない

不一致があった場合は、
- 修正できる → docs を修正
- 即断できない → 未確認事項へ移動

---

### 2.5-2. repo.profile.json ↔ docs の突合（必須）

以下を **双方向** に確認する。

#### repo.profile.json → docs
- commands に定義されたものが、docs のどこかで説明されているか
- doc_roots が docs 構造と一致しているか

#### docs → repo.profile.json
- docs に登場する主要コマンドが、repo.profile.json に存在するか
- 存在しない場合：
  - CI / scripts に実体があるか
  - それも無ければ未確認事項へ

### 2.5-3. CI 定義との整合性確認（必須）
- .github/workflows/**/*.yml に定義されている
  - 実行コマンド
  - ビルド・テスト手順
が docs の記述と矛盾していないかを確認する。

CI を「最優先の事実」として扱い、
docs 側がズレている場合は docs を修正する。

### 2.5-4. 根拠表記の正規化（必須）
docs 内で事実を断定する箇所には、
可能な限り以下の形式で根拠を示す。

- ファイルパス:行番号（または行範囲）
  例：
    - src/server.ts:12-45
    - package.json:scripts.build
    - .github/workflows/ci.yml:jobs.test

根拠が示せない断定は禁止する。

### 2.5-5. 未確認事項の分離（必須）
以下は「失敗」ではない。
未確認のまま混在させることが失敗である。

未確認事項には必ず次を含める：
- 何が未確認か
- なぜ確定できないか
- 何を見れば確定できるか（具体的パス）

### 2.5-6. フェーズ完了条件（Done Criteria）
以下をすべて満たしたときのみ、
/init-docs は完了とする。

- docs に記載された事実が、実体と矛盾していない
- repo.profile.json と docs が相互に説明可能
- CI 定義と docs の手順が一致している
- 未確認事項が明示的に分離されている

これを満たせない場合、
/init-docs は「部分完了」として扱い、
完了宣言をしてはならない。

### 注意
このフェーズは「重い」が正しい。
速度よりも、再現性と信頼性を最優先する。

## Phase 3: AGENTS.md 更新
AGENTS.md を AI 運用の起点として更新する。

必ず以下のルールを維持すること。

    ```md
    ## Custom / Command の使い分け（AI向けルール）

    - init-docs.md: repo の実態把握と設計ドキュメント再構築。重い初期化。
    - docs-sync.md: 実装差分に追随する軽量同期。HARD STOP 時は /init-docs を要求して終了する。
    ```

---

## 出力形式（固定）
- 今回の変更サマリ
- 作成 / 更新ファイル一覧
- 重要な決定事項（根拠付き）
- 未確認事項と次に見るべきファイル

---

## 原則
- このコマンドは「重い」が正しい
- docs-sync では説明不能になった時点で、ここに戻る
- 自動化よりも一貫性と再現性を優先する
