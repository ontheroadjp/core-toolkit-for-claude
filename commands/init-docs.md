# /init-docs

あなたはこのリポジトリの「実態に基づく」プロジェクト設計ドキュメントを作成・更新・管理する AI エージェントです。
想像・憶測は禁止。すべての主張は、必ずリポジトリ内の根拠（ファイルパス、該当行、設定値、実装箇所）に紐づけて説明してください。
不明な点は「未確認」と明示し、追加で何を見れば確定できるかを提示してください。

---

## このコマンドを使うタイミング（再実行トリガー）

以下のいずれかに該当する場合、このコマンドで **再観測・再構築** を行う:

- /docs-sync が HARD STOP を検知した
- docs/.ai/repo.profile.json に載っていないコマンドや doc_root が運用上必要になった
- docs（L1/L2/L3）が現状の構造や責務を説明できなくなった
- 起動方法・主要エントリポイントが変わった疑いがある
- 新しいレイヤ（apps / packages / infra 等）が導入された

---

## 実行前提ゲート（必須）

### G-1: git リポジトリであること
- `.git/` が存在すること
- 存在しない場合: /init-git の実行を促して終了する

---

## 実行方針（最重要）
- まず現状分析 → Repo Profile 生成 → ドキュメント生成 → 整合性検証 → CLAUDE.md 更新
- ドキュメントは「なぜそうなっているか」を必ず説明する
- 冗長な百科事典は禁止。根拠と決定事項を中心に書く
- 生成後、必ず差分サマリ（新規 / 更新 / 保留）を出す

---

## ワークフロー

### Phase 1: プロジェクト分析

すべて根拠付きで列挙すること。確定できないものは未確認事項として分離する。

#### Step 1. ディレクトリ構造の把握
- トップレベルのディレクトリ・ファイル一覧を取得する
- 各ディレクトリの責務を「ファイル名・内容から推定」ではなく「実装から確認」して記述する
- モノレポ構成（apps/ packages/ 等）の有無を確認する

#### Step 2. 技術スタックの特定
以下を優先順位順に確認する:
1. CI 定義（`.github/workflows/*.yml`）— 最優先の事実
2. パッケージ定義（`package.json` / `go.mod` / `Gemfile` / `requirements.txt` / `Cargo.toml` 等）
3. 設定ファイル（`tsconfig.json` / `.eslintrc` / `vite.config.*` 等）

確認項目:
- 言語・ランタイムとバージョン
- パッケージマネージャ（npm / pnpm / yarn / bun / pip / etc.）
- 主要フレームワーク・ライブラリ

#### Step 3. エントリポイントの特定
- `main.*` `server.*` `app.*` `index.*` `pages/` 等を確認する
- CI の実行コマンド（start / dev / build）から逆引きする
- 複数エントリポイントがある場合はすべて列挙する

#### Step 4. 実装済み機能の検出
- ルーティング定義・コントローラ・モデル等から実装済み機能を列挙する
- 「ディレクトリ名から推測」ではなく「ファイルの中身から確認」する

#### Step 5. 主要依存関係の裏取り
- 本番依存（dependencies）と開発依存（devDependencies）を分けて確認する
- バージョンは実際の lock ファイルから取得する（推測しない）

---

### Phase 2: Repo Profile 生成（docs/.ai/repo.profile.json）

- 想像・補完は禁止
- 確定できない項目は出力しない
- CI 定義 > 実行定義 > docs の順で根拠を採用する
- 既存ファイルがある場合は差分更新のみ

最小スキーマ例（参考）:

```json
{
  "repo_id": "string",
  "package_manager": "pnpm|npm|yarn|bun|unknown",
  "doc_roots": [],
  "commands": {}
}
```

---

### Phase 3: ドキュメント生成

以下の構成を基本とし、検出された構成に応じて追加する。

#### docs/L0_concept/（プロダクトコンセプト・設計ポリシー）
生成条件: 常に生成する（全リポジトリ共通）。

- `concept.md`: プロダクトの目的・解決する問題・対象ユーザー・設計上の制約
- `policy.md`: 技術選定ポリシー・セキュリティ方針・パフォーマンス要件・禁止事項

記載ルール:
- 「なぜこの設計か」を記述する（WHY 層）。「何を・どう実装するか」は L1〜L3 に委ねる
- 実体から確認できる事実のみ記述する。推測は禁止
- AI はコード変更の判断前に L0 を参照し、ポリシーと矛盾しないかを確認する

更新トリガー: `/init-docs` 再実行時、または設計方針の根本的変更があった場合。`/docs-sync` での部分更新は行わない（L0 は事実の差分追従ではなく意思決定の記録のため）。

#### docs/L1_project/（プロジェクト全体像）
- `project_overview.md`: プロジェクトの目的・技術スタック・主要機能の一覧
- `repository_structure.md`: ディレクトリ構造と各ディレクトリの責務（根拠付き）

#### docs/L2_development/（開発・運用）
- `operation_model.md`: ローカル起動・ビルド・テストの手順（CI から逆引きして確認）
- `consistency_checks.md`: 整合性確認の観点と手順
- `cicd.md`: CI/CD パイプラインの構成と各ステップの意味（CI 定義がある場合のみ生成）
- `test.md`: テスト戦略・実行方法・カバレッジ方針（テストが存在する場合のみ生成）

#### docs/L3_implementation/（実装仕様）
- `specification_summary.md`: 実装仕様のサマリ（機能・API・データ構造等）
- `database.md`: DB スキーマ・マイグレーション方針（DB が存在する場合のみ生成）
- `api.md`: エンドポイント一覧・リクエスト/レスポンス形式（API が存在する場合のみ生成）

各ドキュメントの記述ルール:
- 事実を断定する箇所には必ず根拠（`ファイルパス:行番号`）を示す
- 「なぜそうなっているか」を必ず添える
- 確認できない事項は未確認として分離する

---

### Phase 4: 整合性検証（必須・保証フェーズ）

このフェーズの目的は、「生成したドキュメントが現時点のソースコード・設定・CI 定義と機械的に矛盾していないことを確認し、保証できない部分を明示的に切り分けること」である。

ここを通過しない限り、/init-docs は「整合性が取れた」と宣言してはならない。

#### 基本原則
- 想像・補完は禁止
- 「合っているはず」は不可
- 確認できないものは **必ず未確認事項として分離**
- 暗黙の前提を docs に残さない

#### 4-1. docs → 実体 の検証（必須）

生成・更新した docs 内から以下をすべて抽出し検証する:

検証対象:
- ファイルパス / ディレクトリパス
- コマンド（run / build / test / dev 等）
- エントリポイント（main, app, server 等）
- 設定ファイル参照（config, env, CI）

検証ルール:
- パス: 実在すること（大文字小文字含む）
- コマンド: docs/.ai/repo.profile.json の commands に存在する、または CI 定義に存在する
- エントリポイント: 実ファイルが存在し、docs の説明と矛盾しない

不一致があった場合:
- 修正できる → docs を修正する
- 即断できない → 未確認事項へ移動する

#### 4-2. docs/.ai/repo.profile.json ↔ docs の突合（必須）

双方向に確認する:

docs/.ai/repo.profile.json → docs:
- commands に定義されたものが docs のどこかで説明されているか
- doc_roots が docs 構造と一致しているか

docs → docs/.ai/repo.profile.json:
- docs に登場する主要コマンドが docs/.ai/repo.profile.json に存在するか
- 存在しない場合: CI / scripts に実体があるか。それも無ければ未確認事項へ

#### 4-3. CI 定義との整合性確認（必須）
- `.github/workflows/**/*.yml` に定義されている実行コマンド・ビルド・テスト手順が docs の記述と矛盾していないかを確認する
- CI を「最優先の事実」として扱い、docs 側がズレている場合は docs を修正する

#### 4-4. 根拠表記の正規化（必須）
docs 内で事実を断定する箇所には可能な限り以下の形式で根拠を示す:
- `src/server.ts:12-45`
- `package.json:scripts.build`
- `.github/workflows/ci.yml:jobs.test`

根拠が示せない断定は禁止する。

#### 4-5. 未確認事項の分離（必須）
未確認のまま混在させることが失敗である。未確認事項には必ず次を含める:
- 何が未確認か
- なぜ確定できないか
- 何を見れば確定できるか（具体的パス）

#### 4-6. フェーズ完了条件（Done Criteria）
以下をすべて満たしたときのみ /init-docs は完了とする:

- docs に記載された事実が実体と矛盾していない
- docs/.ai/repo.profile.json と docs が相互に説明可能
- CI 定義と docs の手順が一致している
- 未確認事項が明示的に分離されている

満たせない場合は「部分完了」として扱い、完了宣言をしてはならない。

このフェーズは「重い」が正しい。速度よりも再現性と信頼性を最優先する。

---

### Phase 5: README.md の検証・scaffold

`~/.config/claude-code-kit/templates/readme.md` を基準として、現在のリポジトリの `README.md` を確認する。

#### 必須セクションの存在確認
以下のセクションが存在しない場合は scaffold（追加）する:
- `## Features` または `## Commands` — アクティブな機能・コマンド一覧
- `## Installation` — セットアップ手順
- `## Usage` — 実行方法・開発コマンド
- `## Design Principles` — 設計制約（Claude がコンテキスト取得に使う）

#### オプションセクション（存在する場合のみ追加）
- `## Architecture` — 構造が複雑なリポジトリのみ
- `## Configuration` — ユーザー向け設定オプションがある場合
- `## Contributing` — コントリビューションフローがある場合
- `## License` — ライセンスが存在する場合

scaffold 後、README.md の内容が実体と矛盾していないことを Phase 4 の観点で検証する。

---

### Phase 6: CLAUDE.md 更新

CLAUDE.md を AI 運用の起点として更新する。
以下のセクションを必ず含め、内容は現時点の各コマンドの実態に合わせて更新する:

```md
## Custom / Command の使い分け（AI向けルール）

- task.md: ドキュメント変更を伴う実装に特化。issue 自動生成〜実装〜ドラフト PR 作成まで。docs/* は変更しない。
- patch.md: ドキュメント変更を伴わない軽微な修正に特化。issue/PR 不要。branch + commit → ユーザーが main へマージ。スコープが広がった場合は /task へエスカレーション。
- docs-sync.md: git diff を事実として docs を最小更新し、ドラフト PR を公開する。HARD STOP 時は /init-docs を要求して終了する。
- init-docs.md: repo の実態把握と設計ドキュメント再構築。重い初期化。docs-sync が説明不能になった時点でここに戻る。
```

Phase 6 完了後、以下の形式で最終報告を行う。

---

## 完了後の報告（出力形式・固定）

- 今回の変更サマリ
- 作成 / 更新ファイル一覧
- 重要な決定事項（根拠付き）
- 未確認事項と次に見るべきファイル
- 完了 / 部分完了 の判定と理由

---

## 原則
- このコマンドは「重い」が正しい
- /docs-sync では説明不能になった時点で、ここに戻る
- 自動化よりも一貫性と再現性を優先する
