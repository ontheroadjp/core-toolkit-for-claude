/init-git

あなたはこのディレクトリを「安全に Git/GitHub 管理下へ初期化」するAIエージェントです。
想像・憶測は禁止。すべての操作は実ファイルの存在確認とユーザーの明示的選択に基づいて行ってください。
不可逆操作は、必ず事前に計画を提示し、ユーザー承認を得てから実行してください。

# 目的
- このディレクトリが Git 管理下にあるかを判定する
- Git/GitHub 初期化を、安全かつ再現可能な手順で完了させる
- ワーキングディレクトリをクリーンな状態にする
- init-docs / init-test / fix 等の前提条件を満たす

# 役割分離（厳守）
- Git 操作（init / branch / commit / push / stash 等）：git
- GitHub 操作（repo 作成 / remote 設定等）：gh
- gh が git の代替になると解釈してはいけない

# 前提
- 作業ディレクトリはリポジトリルートである
- git コマンドが利用可能である
- gh コマンドが利用可能である
- ネットワークエラー等が発生した場合は、処理を停止し「サンドボックス外で実行すべき手順」を提示する

# 追加前提（必須ゲート）
1) docs/ の存在確認
- docs/ が存在しない場合は、このコマンドを開始してはいけない
- ユーザーに /docs-init の実行を促して停止する（他の作業はしない）

2) repo.profile.json の存在確認（.gitignore 更新のため）
- repo.profile.json が存在しない場合は、このコマンドを開始してはいけない
- ユーザーに /init-docs の実行を促して停止する（他の作業はしない）

# 0. 事前ゲート（必須・最初に実行）
以下を必ず最初に確認し、条件を満たさない場合は直ちに停止する。

## G-1: docs/ が存在するか
- docs/ が存在しない場合：
  - 「docs/ が無いため /init-git は実行しない」と明示する
  - /docs-init の実行を促して終了する

## G-2: repo.profile.json が存在するか
- repo.profile.json が存在しない場合：
  - 「repo.profile.json が無いため .gitignore を現状に即して維持できない」と明示する
  - /init-docs の実行を促して終了する

# 1. Git 状態の判定
- `.git/` ディレクトリの有無を確認する

## ケースA: `.git/` が存在しない場合
- Git 未初期化として扱う

## ケースB: `.git/` が存在する場合
- Git 初期化済みとして扱う

---

# 共通: .gitignore を現状に即して維持する（必須）
.gitignore は「新規作成の場合も既存の場合も」常に更新対象とする。

## ルール
- repo.profile.json を読み取り、使用言語/実行環境が設定されている場合は使用言語/実行環境に基づいて必要な除外を含める
- OS 固有の一般的な除外を含める（mac: .DS_Store 等）
- 既存 .gitignore は尊重し、上書きではなく差分追加/整理を行う
- 想像で言語を決めない（repo.profile.json から確定できない場合は未確認として停止）

## 実装方針（禁止事項）
- 「とりあえず全部盛り」の巨大 .gitignore を作らない
- repo.profile.json で確定できない言語の除外を勝手に追加しない
- 既存ルールの削除は原則しない（明確に不要と根拠がある場合のみ）

---

# ケースA: `.git/` が存在しない場合の処理

## A-1. ローカル事前準備
- `.gitignore` を上記ルールで作成/更新する
- `README.md` が存在しない場合：
  - ローカルに新規作成する
  - 内容は repo 名のみとする

## A-2. git 初期化
- `git init` を実行する
- 初期ブランチ名が master の場合：
  - main にリネームする
- 初期ブランチ名が main の場合：
  - 何もしない

## A-3. 初期コミット
- main ブランチで既存ファイルをステージ
- `git commit -m "initial commit"` を実行する

## A-4. GitHub リポジトリ作成（gh）
以下をユーザーに確認する：

- owner（個人 / org）
- リポジトリ名（デフォルト：ディレクトリ名）
- public / private
- 接続方式（ssh / https）
- GitHub 側で README / .gitignore を作成しないことを明示

承認後、`gh repo create` を実行する。

## A-5. push
- remote が正しく設定されたことを確認する
- 以下を実行する：
  - `git push -u origin main`

---

# ケースB: `.git/` が存在する場合の処理

## B-1. 未コミット変更の検出
- `git status --porcelain` で未コミット変更を検出する
- 変更が無い場合：
  - 何もせず次へ進む（.gitignore 更新は既に実施済み）

## B-2. 変更内容の調査
- 変更ファイル一覧（追加/変更/削除）を提示
- 差分概要を要約する

## B-3. リスクスキャン（固定基準）
以下に該当する可能性を検出・提示する：

- 秘密情報の可能性
  - .env*, *.pem, id_rsa*, credentials*, secret*, apikey*
  - 文字列パターン（例：BEGIN PRIVATE KEY, *_SECRET_*）
- 生成物・巨大ファイル
  - dist/, build/, .next/, node_modules/, __pycache__/
  - サイズ閾値超過ファイル（例：5MB以上）
- ローカル依存設定
  - .vscode/, .idea/, *.local, config.local.*

## B-4. ユーザー選択（必須）
以下を提示し、選択を待つ：

- commit
- stash
- 中断

※ 破棄はデフォルトでは提示しない
※ ユーザーが明示的に「破棄も選択肢に含める」と指示した場合のみ提示可

## B-5. commit を選択した場合
- コミット粒度をユーザーに選ばせる：
  - 全変更を1コミット
  - 指定ファイルのみ
  - AI が分割案を提示 → ユーザー承認後に実行
- コミットメッセージはユーザー確認を必須とする
- commit は dev ブランチに行う

## B-6. stash を選択した場合
- `git stash push` を実行する
- stash 名を明示する

# 実行計画（Plan）提示ルール
- 実行前に、以下を必ず一覧で提示する：
  - 実行予定コマンド
  - 作成/変更されるファイル（.gitignore / README.md 等）
  - GitHub 上で行われる操作
- ユーザーの明示的な「実行してよい」回答があるまで、何も実行しない

---

# 出力形式（固定）
A. 事前ゲート結果（docs/ と repo.profile.json）
B. Git 状態判定結果
C. .gitignore 更新内容（追加/変更点）
D. 実行計画（Plan）
E. ユーザー選択内容
F. 実行した操作
G. 現在のブランチ/状態
H. 未確認事項（ある場合のみ）

---

# 絶対にやってはいけないこと
- 事前ゲート未達の状態で処理を続行する
- repo.profile.json 不在で .gitignore を推測更新する
- ユーザー確認なしで不可逆操作を行う
- 既存 .git を破壊・上書きする
- AI判断のみで commit 粒度を確定する
- GitHub 側で README / .gitignore を自動生成させる
- ローカルと GitHub で状態が乖離したまま push する
